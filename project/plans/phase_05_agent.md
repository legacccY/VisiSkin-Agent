# 阶段五：Agent 系统

## 目标

用 ReAct 模式编排整个交互流程：LLM 作为大脑，调用质量评估/分割/分类工具，在照片质量不足时主动追问用户，引导重拍，再进行分析，最终给出诊断建议。

---

## 前置条件

- 阶段三完成：VisiScore-Net 可推理
- 阶段四完成：QAD 模块可推理
- 已确认 LLM 选型（Qwen3-8B 4-bit 量化可在 RTX 4070 运行）

---

## 本阶段工具

| 工具 | 用途 |
|------|------|
| bitsandbytes | Qwen3-8B 4-bit 量化（把 8B 模型压到 ~4.5GB 显存） |
| accelerate | 优化模型加载，管理 CPU/GPU 设备映射 |
| gradio | 搭建图片上传 + 多轮对话 Demo UI |

---

## 关键技术决策点

1. **LLM 选择**：
   - Qwen3-8B 4-bit 量化（~4.5GB 显存，能力强，推荐）
   - Qwen3-4B 4-bit 量化（~2.5GB 显存，留更多余量给视觉模型）
   决策标准：与视觉模型共存时总显存 ≤ 7.5GB

2. **最大交互轮次限制**：3 轮 / 5 轮 / 不限。影响用户体验和误用风险

3. **是否支持语音输入**：否（MVP 阶段先不做）/ 是（接 whisper 转录）

---

## 工作步骤

1. **工具定义**：将阶段三/四的模型封装成 Agent 可调用的工具函数，定义标准化输入输出
2. **ReAct 状态机**：实现 Thought → Action → Observation 循环，定义终止条件（质量达标或轮次上限）
3. **问题库建设**：整理皮肤科临床常见追问（部位/时长/变化趋势/痒痛感），按质量问题类型分类
4. **Gradio Demo**：搭建图片上传 + 对话界面，测试完整交互流程
5. **端到端评测**：在测试样本上运行 Agent，统计引导成功率和诊断准确率

---

## 验收标准

- Gradio Demo 可正常启动，上传低质量图片后 Agent 正确触发追问
- 上传高质量图片后 Agent 直接给出诊断（不无谓追问）
- 整个推理链路端到端响应 <3s（首次加载模型后）
- `eval_agent.py` 输出包含：平均交互轮次、引导成功率、最终诊断准确率

---

## 硬件/资源约束

- LLM + 视觉模型同时加载，总显存需控制在 7.5GB 以内
- 建议延迟加载（用户上传图片时才加载视觉模型），避免启动时 OOM
- 如果显存紧张，可将 LLM 和视觉模型分时加载（不同步骤用不同模型）

---

## 注意事项

- ReAct 循环中 LLM 的 system prompt 非常关键，需要精心设计，限制 LLM 只调用已定义工具
- 问题库的追问措辞要口语化、非医疗术语，适合普通用户理解
- `ask_user` 工具在 Gradio 中体现为 chatbot 的等待用户输入，需要处理异步状态
- Agent 不能给出确定性诊断（法律风险），措辞应为"建议就医"而非"确诊为XXX"
- 论文实验中，Agent 的追问策略是核心创新之一，要记录每次追问的触发条件和用户响应
